# Companion-Agent v5 — 需求文档 / Codex 提示词

> 本版把《开源轮子评估报告》的结论落地。主题:**记忆与关系深化**。核心三件事——记忆层接入 Mem0、角色关系状态激活、角色定义结构化(角色卡)。延续 v3/v4 的目录与接口边界规约。整份交给 Codex。文末第 10 节列出需人类确认/准备的部分。

---

## 0. 本版定位

v4 已把"标准家具 + 推送地基 + 管家 dock + 主动性"做完。v5 不加新界面模块,而是**把已有人格/记忆从"能跑"提升到"有深度、长期一致"**,用调研中验证过的轮子和模式:Mem0(记忆)、SillyTavern 的关系/场景状态追踪模式(关系)、角色卡 CCv3 模式(人格定义)。

**明确不动(已是正确选择,本版不要重构):**
- 聊天 UI:`chatscope`(继续用)。
- 模型网关:`LiteLLM`(继续用)。
- 编排层:薄自建(路由→扇出→工具循环),**不引入 LangGraph/CrewAI 等重框架**。
- 工具侧:保留 v4 自建 SQLite 实现(见 §6,Vikunja 仅作接口后可选升级,本版不强制替换)。

---

## 1. 记忆层:接入 Mem0(替换纯文本 notebook)

把 v3 留好的 `MemoryStore` 接口的实现,从"纯文本 notebook"换成 **Mem0 开源自托管**。原因:Mem0 写入路径原生在 ADD/UPDATE/DELETE/NOOP 间选择,天然支持"旧事实被新事实覆盖改写"(正是兄弟把"要去健身"改成"跑了 PB"的语义),且自动从对话抽取要点,省去手写抽取逻辑。

### 1.1 接入方式
- 用 **Mem0 开源 SDK + 本地向量存储**(如 Qdrant/Chroma 本地实例),**不接 Mem0 云、不使用付费图谱层**。开源向量层 + 自动抽取对单用户足够。
- 每个角色(persona)对应 Mem0 的一个独立 memory scope(用 agent_id / user 维度隔离),保持"每角色独立记忆"的产品语义不变。
- 管家的"事实账本"仍走工具侧(todos/memos),**不混入** Mem0;Mem0 只承载角色的关系型记忆。

### 1.2 两条反锁定铁律(必须遵守)
1. **原始事件流存自己的库,Mem0 只当索引。** 所有对话原文继续存在你的 SQLite `messages` 表;Mem0 里存的是从对话抽取出的记忆条目。这样将来换记忆框架是"重新灌一遍",不是"从框架抢救数据"。
2. **抽取用便宜模型 + 不每条都抽。** Mem0 写入会调 LLM 抽取,固定走 `MODEL_CHEAP`;批量/按需写入,避免每条消息都触发一次抽取调用(成本护栏)。

### 1.3 迁移
- 把现有 `persona_notes` 里的条目一次性导入对应角色的 Mem0 scope(写个一次性迁移脚本)。迁移后 `persona_notes` 表可保留只读备份,不再写入。
- 角色生成回复时,从 Mem0 检索该角色相关记忆注入 prompt(替换原来全量注入 notebook 的做法)。

### 1.4 时机说明
现在开始 dogfooding、记忆会快速增长,故本版即接入(而非继续等)。但务必先把 §1.2 铁律落到位,否则将来难回退。

**验收**:兄弟说"要去健身"→ Mem0 出现一条相关记忆;后续"跑了 PB"→ 该记忆被 UPDATE/替换而非新增并存;换个话题几十轮后再问相关事,角色仍能检索到。

## 2. 角色关系状态激活(SillyTavern 关系追踪模式)

v3/v4 已留 `persona_state` 表位,本版填实。借鉴 SillyTavern 的关系/数值追踪扩展:维护每个角色与用户之间的关系状态,并注入 prompt 影响语气。

### 2.1 字段(每角色一行)
- `familiarity`:熟悉度(数值,随互动累积、随长期不互动缓慢衰减)。
- `last_tone`:最近一次互动的基调(如 轻松/认真/疏远)。
- `last_interaction_at`:上次互动时间(配合 v4 时间感:多日未互动,角色可主动表达"好久没聊")。
- `milestones`:关系里程碑列表(自由文本,如"一起定下了健身计划""分享过研究焦虑")。

### 2.2 演变规则(本版给保守默认,规则可后续迭代)
- 每次角色与用户互动后,更新 `last_tone` / `last_interaction_at`,`familiarity` 小幅 +。
- 长期(如多日)无互动,`familiarity` 缓慢衰减。
- 里程碑的提炼:可复用 §1 的 Mem0 抽取结果中"重要事件"类条目,**不另起一套抽取**。
- 注入:把关系状态摘要并入角色 prompt 的上下文,让语气随熟悉度/基调变化。

> 具体的衰减速率、familiarity 如何精确影响语气,属手感项,见第 10 节;本版做成可配 + 合理默认。

**验收**:连续多天与兄弟互动,其语气体现"更熟";隔多日再聊,他能感知并表达"有阵子没聊了";关系里程碑可在角色记忆/状态里看到。

## 3. 角色定义结构化(角色卡 CCv3 模式)

把角色从"一段扁平 system prompt"升级为**结构化角色卡**,提升长期一致性(借鉴 SillyTavern CCv3 的字段组织,不引入其代码)。

- 角色卡字段:`persona_core`(核心设定)、`speaking_style`(语气/长短/口头禅)、`example_dialogues`(若干示例对话,锚定说话风格)、`world_info`(与用户共享的背景设定,可选)。
- 后端在调角色模型时,把角色卡按固定模板拼成 system prompt + few-shot 示例。
- 这些字段存 `personas`(或新 `persona_card` 表);**可在设置面板编辑**(用户能调教角色)。

**验收**:角色说话风格在长对话中更稳定、更贴人设;在设置里改角色卡后,角色行为随之变化。

---

## 4. 场景/时间状态深化(可选,基于 v4 时间感)

在 v4 `time_context` 基础上,借鉴 SillyTavern 场景状态追踪,可选地维护轻量"当前场景上下文"(如:最近话题、用户当前状态)。本版**仅做接口/字段留位 + 最小实现**,避免过度设计;深做与否看 dogfooding 反馈。

---

## 5. 数据模型变更(相对 v4)

```
persona_state(填实 v3/v4 留位)
  persona_id          INTEGER FK
  familiarity         REAL DEFAULT 0
  last_tone           TEXT NULL
  last_interaction_at TEXT NULL
  milestones          TEXT NULL          -- JSON 数组

persona_card(新, 或并入 personas)
  persona_id          INTEGER FK
  persona_core        TEXT
  speaking_style      TEXT
  example_dialogues   TEXT               -- JSON
  world_info          TEXT NULL

# 记忆:persona_notes 转为只读备份;新记忆走 Mem0(外部 store,不在主表)
# settings 增加:mem0 配置、familiarity 衰减/影响参数(集中于 core/config)
```

接口侧:
- `MemoryStore` 新增 Mem0 实现(`add/search/update`),替换 notebook 实现;调用方(角色回复逻辑)不变。
- 关系状态读写封装为 `core` 的小工具,供角色回复后统一更新。

保留 v4 自建 SQLite 的 `ToolStore` 实现。Vikunja/Super Productivity 作为"接口后可替换"的升级项**记录在案但本版不做**——理由:v4 的 todo 已能用,现在替换要多运维一个服务且需迁移,边际收益不值当;`ToolStore` 接口已留好,将来想换随时可换。

## 7. 反屎山规约(沿用 v3/v4)

- 三铁律不变:配置驱动零硬编码、边界即接口、目录隔离。
- **记忆层换实现不得波及调用方**:角色回复逻辑只认 `MemoryStore` 接口,Mem0 细节封死在实现类里。
- Mem0 配置(store 类型、抽取模型、scope 规则)集中在 `core/config.py`。
- 关系状态演变逻辑集中一处,不散落到各角色代码。

---

## 8. 构建顺序

1. **Mem0 接入**:装 SDK + 本地向量 store;实现 `MemoryStore` 的 Mem0 版;落实两条铁律;迁移 `persona_notes`;切换角色记忆注入。跑 §1 验收。
2. **关系状态**:填实 `persona_state` + 演变规则 + 注入。跑 §2 验收。
3. **角色卡结构化**:`persona_card` + 模板拼装 + 设置面板编辑。跑 §3 验收。
4. **场景状态(可选最小实现)**。

每步跑回归:v2"健身→pb"、v4 时间感与主动性,必须始终通。

---

## 9. 性能/成本注意

记忆检索(向量查询)与抽取(LLM 调用)都在角色回复路径上,注意:检索走本地向量 store(快);抽取走便宜模型且不每条都抽。监控单回合的总调用数,别让"记忆深化"把延迟和成本顶上去。

---

## 10. 需人类确认 / 准备

**需你准备:**
- **确认在云 Linux 服务器上跑本地向量 store 的资源**(Qdrant/Chroma 占用不大,但确认磁盘/内存够)。
- **角色卡内容**:若想要更鲜活的兄弟/老师/管家,可由你提供各角色的核心设定、语气、几条示例对话(这是内容,不是代码,你最懂你想要的角色)。本版可先用占位默认,你随时在设置里改。

**需你后续拍板(本版先给默认):**
- familiarity 的衰减速率与对语气的影响强度(手感,dogfooding 校准)。
- 关系里程碑提炼到什么粒度。
- 场景状态是否值得深做。
- 工具侧是否最终换 Vikunja(本版不动)。

> 其余按本文档实现。任何破坏接口边界的"省事写法"一律不采用。
