# 项目架构总结

本文档总结当前项目的实际代码架构、核心模块职责、数据流和运行方式。项目整体是一个本地运行的 AI 伙伴/聊天群应用，包含多角色聊天、管家待办、备忘录、习惯、关系状态和角色卡管理。

## 1. 总体架构

项目采用前后端分离架构：

- 前端：`frontend/`，Vite + React 单页应用，负责聊天界面、角色管理、待办/记忆侧栏、主题设置和管家 Dock。
- 后端：`backend/`，FastAPI 应用，负责 REST API、SSE 实时事件、SQLite 数据持久化、LLM 调用、角色记忆、关系状态和管家工具执行。
- 数据库：项目根目录 `app.db`，SQLite 文件，由 SQLModel 建表和轻量迁移。
- 模型调用：后端通过 `litellm` 统一调用模型，模型与功能开关存储在 `settings` 表，也可以由 `.env` 提供 API Key。
- 归档文件：`unused_files_20260626/` 存放历史规格、构建产物和压缩包，不属于当前运行路径。

典型运行路径：

1. 用户在 React 前端输入消息。
2. 前端调用 `/api/channels/{channel_id}/messages`。
3. FastAPI 路由转交 `ChatService.handle_user_message()`。
4. 后端保存用户消息，触发 SSE 推送。
5. 聊天角色调用 `run_persona_agent()` 生成回复，并可通过工具维护私有记忆。
6. 管家 `StewardService` 静默分析消息，必要时创建/更新待办、备忘录、习惯记录。
7. 后端保存 AI 回复和账本变化，通过 SSE 推送到前端。
8. 前端刷新消息、待办、备忘录、关系状态等视图。

## 3. 后端架构

后端基于 FastAPI，入口为 `backend/main.py`。

### 3.1 应用入口

`backend/main.py` 负责：

- 创建 `FastAPI(title="Companion Agent v3")`。
- 配置 CORS，允许本地 Vite 前端访问。
- 启动时初始化数据库：`init_db()`。
- 启动时写入种子数据：`seed_data()`。
- 启动后台管家心跳任务：`steward_heartbeat()`。
- 挂载 `/api` 路由。
- 如果存在 `frontend/dist`，则将构建后的前端作为静态站点挂载到 `/`。

### 3.2 API 路由层

`backend/api/routes.py` 是主要 REST API 集中入口，统一挂载在 `/api` 下。

主要 API 分组：

- 角色：`/personas`、`/personas/{id}`、`/personas/{id}/card`、`/personas/{id}/notes`
- 频道：`/channels`、`/channels/{id}`、`/channels/{id}/messages`
- 实时事件：`/channels/{id}/events`
- 待办：`/todos`
- 备忘录：`/memos`
- 习惯：`/habits`
- 关系：`/relations`
- 日程：`/schedule`
- 管家：`/steward/brief`、`/steward/messages`、`/steward/proactivity/tick`
- 设置：`/settings`、`/settings/{key}`

路由层本身主要做参数转换、实体检查和响应拼装，核心业务委托给 service、agent 和 store 模块。

### 3.3 数据层

`backend/models.py` 使用 SQLModel 定义数据库表：

- `Persona`：角色基础信息、系统提示词、模型角色、模型覆盖配置、模拟输入配置。
- `PersonaCard`：角色卡，包括核心设定、说话风格、示例对话、世界信息、声音、性格标签。
- `PersonaState`：关系状态，包括熟悉度、最近语气、上次互动、里程碑。
- `PersonaNote`：每个角色私有记忆。
- `Channel`：聊天频道，支持私聊、群聊、管家系统频道。
- `ChannelMember`：频道成员关系。
- `Message`：消息记录。
- `Todo`：待办事项。
- `Memo`：备忘录。
- `Habit` / `HabitLog`：习惯与习惯日志。
- `Setting`：应用配置和模型配置。

`backend/db.py` 负责：

- 设置 SQLite 数据库路径：`app.db`。
- 创建 SQLModel engine。
- `init_db()` 创建表。
- `migrate_sqlite()` 对旧表结构做增量字段补齐。
- `session_scope()` 提供会话上下文。

### 3.4 种子数据和默认配置

`backend/seed.py` 在启动时写入默认角色和频道：

- `兄弟`：口语、调侃型 AI 伙伴。
- `老师`：严谨、克制、方法论型 AI 伙伴。
- `管家`：系统角色，默认后台维护待办和备忘录。

同时会创建：

- 默认 `兄弟` 私聊频道。
- 默认 `管家` 系统频道。
- 默认角色卡。

`backend/core/config.py` 负责默认设置：

- 模型配置：`model.chat_strong`、`model.chat_cheap`、`model.steward`。
- 模型目录：`model.catalog.sjtu`。
- 模拟输入效果：`sim.*`。
- 管家主动提醒：`proactivity.*`。
- 记忆后端：`memory.*`。
- 关系系统：`relationship.*`。

### 3.5 聊天服务

`backend/chat/service.py` 是普通聊天的主业务服务。

核心职责：

- 创建、读取、更新、删除频道。
- 查询频道成员和消息。
- 保存用户消息。
- 根据频道类型决定回复角色：
  - 私聊：该频道唯一角色回复。
  - 群聊：如果消息 `@角色名`，则被提及角色回复；否则所有成员回复。
  - 管家系统频道：走管家 Dock 回复。
- 调用角色 Agent 生成回复。
- 应用角色记忆工具调用。
- 更新关系状态。
- 触发管家后台分析。
- 将回复按段拆分，模拟输入延迟，并通过 SSE 推送。

### 3.6 角色 Agent

`backend/chat/agents.py` 负责普通角色的 LLM 调用。

输入上下文包括：

- 角色卡：核心设定、说话风格、示例对话、世界信息。
- 检索到的私有记忆。
- 当前关系状态。
- 当前时间上下文。
- 最近频道消息。

角色 Agent 可调用三个记忆工具：

- `add_note`
- `update_note`
- `delete_note`

当没有可用模型或强制 fallback 时，代码会使用 `fallback_persona()` 返回固定规则回复，保证本地没有 API Key 时仍能跑通基础流程。

### 3.7 记忆系统

`backend/chat/memory.py` 提供两种记忆实现：

- `SQLiteMemoryStore`：直接使用 `persona_notes` 表检索和保存角色私有记忆。
- `Mem0MemoryStore`：优先使用 mem0 + Chroma 向量存储，同时保留 SQLite 作为兼容缓存。

`build_memory_store()` 根据 `settings.memory.backend` 决定使用哪种后端。当前默认配置为 `mem0`，失败时会回退到 SQLite 检索。

### 3.8 关系状态

`backend/chat/relationship.py` 负责维护角色与用户的关系状态：

- 每轮有效互动增加熟悉度。
- 根据距离上次互动的天数衰减熟悉度。
- 根据用户文本推断最近语气。
- 将重要内容合并到里程碑列表。

这些信息会在角色 Agent prompt 中作为关系上下文输入。

### 3.9 管家系统

管家逻辑由 `backend/steward/service.py` 和 `backend/steward/agent.py` 组成。

`StewardService` 负责：

- 在用户聊天后静默运行管家分析。
- 在管家 Dock 中直接回复用户。
- 应用管家工具调用到 SQLite 账本。
- 主动提醒 `proactivity_tick()`。

`run_steward_agent()` 可调用工具：

- `create_todo`
- `update_todo`
- `complete_todo`
- `write_memo`
- `list_todos`
- `upsert_habit`
- `log_habit`

管家和普通角色不同：管家默认不输出聊天内容，主要维护客观账本；只有在管家 Dock 频道中才生成用户可见回复。

### 3.10 实时通信

`backend/core/transport.py` 实现 `SSETransport`。

事件机制：

- 后端保存用户消息或 AI 回复后调用 `transport.push_nowait()`。
- 前端通过 `EventSource('/api/channels/{id}/events')` 订阅。
- 支持事件类型：
  - `ready`
  - `typing`
  - `message`
  - `proactive`

这让消息和输入状态可以实时出现在前端，不需要轮询。

### 3.11 LLM 封装

`backend/core/llm.py` 使用 `litellm.completion()` 做模型调用。

核心函数：

- `provider_ready(model)`：判断是否具备可用模型和 API Key。
- `run_tool_loop()`：执行带工具调用的多轮 LLM 循环。

支持的 API Key 环境变量包括：

- `OPENAI_API_KEY`
- `DEEPSEEK_API_KEY`
- `ANTHROPIC_API_KEY`
- `OPENROUTER_API_KEY`
- `MOONSHOT_API_KEY`
- `ZHIPUAI_API_KEY`

如果设置 `LLM_FORCE_FALLBACK=1`，后端会跳过真实模型调用，使用 fallback 逻辑。

## 4. 前端架构

前端位于 `frontend/`，使用 Vite + React。

### 4.1 主要文件

```text
frontend/
├── index.html
├── package.json
├── vite.config.js
└── src/
    ├── App.jsx        # 主应用、状态、API 调用和大部分组件
    └── styles.css     # 全局样式、主题、布局和响应式规则
```

### 4.2 前端状态组织

`frontend/src/App.jsx` 是当前前端核心，采用函数组件和 React hooks 管理状态。

主要状态包括：

- UI 状态：主题、强调色、侧栏折叠、当前导航页、弹窗状态。
- 聊天状态：频道、当前频道、消息、输入框、发送状态、输入中状态。
- 管家状态：管家摘要、管家消息、管家 Dock 展开状态。
- 账本状态：待办、备忘录、习惯、关系。
- 角色状态：角色列表、当前角色、新建角色草稿。
- 设置状态：模型、记忆、主动提醒等配置。

### 4.3 API 调用方式

前端统一通过：

```js
const API = "/api";
async function request(path, options = {}) { ... }
```

访问后端 API。Vite 开发环境下通常通过 dev server proxy 或后端 CORS 与 `127.0.0.1:8000` 配合；生产构建后可由 FastAPI 直接挂载 `frontend/dist`。

### 4.4 页面和组件

当前前端没有拆成多文件组件，主要组件都定义在 `App.jsx` 中。

主要界面：

- 左侧栏：频道列表、角色列表、新建频道/角色入口。
- 聊天主区：消息列表、日期分隔、输入框、发送按钮、清空频道。
- 右侧信息栏：今日概览、待办、记忆/习惯/关系。
- 角色页：角色核心设定、风格、声音、标签、关系熟悉度和统计。
- 管家 Dock：底部管家聊天与快捷指令。
- 设置面板：主题色、模型配置展示、角色页跳转。
- 新建频道弹窗。
- 新建角色弹窗。

### 4.5 样式系统

`frontend/src/styles.css` 定义：

- 基于 CSS 变量的浅色/深色主题。
- 多套强调色：`graphite`、`slate`、`sage`、`clay`。
- 两栏/三栏应用布局。
- 聊天气泡、频道列表、待办卡片、关系进度、管家 Dock。
- 响应式规则：窄屏隐藏右侧面板和部分顶部操作。

## 5. 核心数据流

### 5.1 普通聊天数据流

```text
用户输入
  ↓
frontend request(POST /api/channels/{id}/messages)
  ↓
routes.post_message()
  ↓
ChatService.handle_user_message()
  ↓
保存用户 Message
  ↓
SSE 推送用户消息
  ↓
选择回复角色
  ↓
memory.search()
  ↓
run_persona_agent()
  ↓
memory.apply_tool_calls()
  ↓
update_relationship_state()
  ↓
persist_persona_reply()
  ↓
SSE 推送 typing/message
  ↓
StewardService.run_for_user_message()
  ↓
管家按需更新 todos/memos/habits
```

### 5.2 管家 Dock 数据流

```text
用户在管家 Dock 输入
  ↓
POST /api/channels/{steward_id}/messages
  ↓
ChatService 检测 system steward channel
  ↓
StewardService.run_dock_message()
  ↓
run_steward_agent() 分析并调用工具
  ↓
SQLiteToolStore 更新账本
  ↓
dock_reply_text() 生成可见回复
  ↓
保存管家 Message 并通过 SSE 推送
```

### 5.3 主动提醒数据流

```text
FastAPI startup
  ↓
steward_heartbeat() 后台循环
  ↓
读取 settings: proactivity.enabled / interval_minutes
  ↓
StewardService.proactivity_tick()
  ↓
查找带 due_time 的 pending todos
  ↓
写入管家系统频道 Message
  ↓
SSE proactive 事件推送
```

### 6.1 后端依赖


## 7. 配置和环境变量

`.env` 会由 `backend/core/config.py` 在启动时加载。

常用配置：

- 模型 API Key：例如 `OPENAI_API_KEY`、`DEEPSEEK_API_KEY` 等。

## 8. 当前架构特点

### 优点

- 前后端边界清晰，API 都集中在 `/api`。
- 后端服务层已经区分聊天、管家、记忆、关系、工具存储。
- SQLite 单文件数据库适合本地个人应用。
- SSE 比轮询更适合聊天消息和输入状态。
- LLM 调用通过 `litellm` 抽象，方便切换模型。
- 没有 API Key 时仍有 fallback，便于本地开发和演示。

### 当前限制

- 前端组件集中在一个很大的 `App.jsx`，后续维护成本会逐渐升高。
- API 路由集中在一个较大的 `routes.py`，后续可以按 personas/channels/todos/settings 拆分。
- 数据库迁移是手写轻量迁移，适合小项目，但长期可考虑 Alembic。
- 当前缺少自动化测试目录。
- 角色 Agent、管家 Agent 和工具 schema 还在代码内硬编码，未来可抽成配置或独立模块。
- `unused_files_20260626/` 已归档但仍在仓库内，构建和搜索时需要注意排除。

## 9. 建议的后续演进方向

- 将 `backend/api/routes.py` 拆成多个路由模块。
- 增加基础测试：
  - API 路由测试。
  - `ChatService` 单元测试。
  - 管家工具调用测试。
  - SQLite 迁移测试。
- 添加项目级 `README.md`，将运行方式和环境变量说明放在更醒目的位置。

中期可做：

- 引入正式迁移工具。
- 为 LLM 工具调用增加更明确的输入校验和错误日志。
